# Infrastructure Layer Context

## Purpose

The infrastructure layer implements interfaces defined in the domain layer. It handles persistence, external APIs, and other infrastructure concerns.

## Architecture Principles

- **Implement domain interfaces**: All repositories implement interfaces from `domain/repository/`
- **Depend on domain**: Infrastructure depends on domain, never the reverse
- **Separate by entity**: Each entity gets its own subdirectory (e.g., `activityRecord/`)
- **Database abstraction**: Use GORM for database operations

## Layer Components

- **`{entity}/`**: Repository implementations (CmdRepo and QueryRepo) for each entity
- **`db/`**: Database services and utilities
- **`db/model/`**: Database models (GORM structs)
- **`helper/`**: External integrations and utilities (if present)
- **`envs/`**: Environment variable constants (if present)

## Repository Implementation Pattern

Each entity directory contains:

- `{entity}CmdRepo.go`: Command repository implementation (Create, Update, Delete)
- `{entity}QueryRepo.go`: Query repository implementation (Read)
- Tests for both repositories

## Package Naming

- Package name: `tkInfra{Entity}` (e.g., `tkInfraActivityRecord`)
- Matches directory structure for clarity

## Database Operations

- Use GORM for database access
- Database models in `db/model/` are separate from domain entities
- Transform models to entities via `model.ToEntity()`
- Transform DTOs to models in repository implementations

## Error Handling

- Return GORM errors directly or wrap them with context
- Log infrastructure errors appropriately
- Never expose database implementation details to domain layer

## Testing Requirements

- MUST test all repository operations
- Use test database or mocks
- Test both success and error scenarios
- Verify model-to-entity transformations
