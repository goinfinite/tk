# Presentation Layer Context

## Purpose

The presentation layer handles user interactions, HTTP requests, and API responses. It validates untrusted input and delegates business logic to use cases.

## Architecture Principles

- **Validate untrusted input**: All external input must be validated before reaching domain
- **Use liaisons**: Bridge between presentation and domain (API/UI → Liaison → Use Case)
- **Transform responses**: Convert domain entities to API/UI-appropriate formats
- **Stateless**: Presentation layer should not maintain business state

## Layer Components (when present)

- **`api/`**: REST API controllers and routing
- **`cli/`**: Command-line interface
- **`ui/`**: Web interface
- **`liaison/`**: Bridge between presentation sub-layers and use cases
- **`middleware/`**: Request processing (auth, logging, panic handling)
- **`init/`**: Shared initialization for presentation sub-layers

## Liaison Pattern

Liaisons handle:

1. Untrusted input validation
2. Repository instantiation
3. Use case invocation
4. Response transformation

```go
func ActionEntityLiaison(
    untrustedInput map[string]any,
) (liaisonResponse LiaisonResponse) {
    // 1. Validate and parse input
    // 2. Create repositories
    // 3. Call use case
    // 4. Transform to LiaisonResponse
    return liaisonResponse
}
```

## Middleware Responsibilities

- Authentication and authorization
- Request logging
- Panic handling and recovery
- Database connection injection
- CORS and security headers

## Error Handling

- Return user-friendly error messages
- Use `LiaisonResponse` for consistent API responses
- Log detailed errors server-side
- Mask sensitive information from untrusted clients

## Package Naming

- Package: `tkPresentation{Component}` (e.g., `tkPresentationMiddleware`)
- Matches directory structure

## Testing

- Test input validation thoroughly
- Test error handling for invalid input
- Test response transformation
- Integration tests for API endpoints
