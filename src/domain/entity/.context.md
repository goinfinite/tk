# Entity Context

## Purpose

Entities represent persistent business objects with unique identities. Unlike value objects, entities have continuity through time and are distinguished by their ID, not their attributes.

## Critical Rules

- **Identity**: All entities have a unique identifier (e.g., `RecordId`)
- **Immutability**: Prefer immutable entities; use constructors to create modified copies
- **Use value objects**: All entity fields should use domain value objects
- **No persistence logic**: Entities don't know how they're stored (repository responsibility)

## Entity Structure

```go
type EntityName struct {
    EntityId          tkValueObject.EntityId       `json:"entityId"`
    RequiredField     tkValueObject.SomeType       `json:"requiredField"`
    OptionalField     *tkValueObject.SomeType      `json:"optionalField"`
    CreatedAt         tkValueObject.UnixTime       `json:"createdAt"`
    UpdatedAt         *tkValueObject.UnixTime      `json:"updatedAt"` // If applicable
}
```

## Constructor Pattern

All entities MUST provide a constructor function unless there are many optional fields, in which case creating the struct directly is preferred to avoid "nil, nil, nil, nil" constructor calls.

```go
func NewEntityName(
    entityId tkValueObject.EntityId,
    requiredField tkValueObject.SomeType,
    optionalField *tkValueObject.SomeType,
    createdAt tkValueObject.UnixTime,
) EntityName {
    return EntityName{
        EntityId:      entityId,
        RequiredField: requiredField,
        OptionalField: optionalField,
        CreatedAt:     createdAt,
    }
}
```

## Field Ordering

1. ID field first
2. Required fields (non-pointer)
3. Optional fields (pointer)
4. Timestamp fields last (CreatedAt, UpdatedAt)

## Common Patterns

- Entities are passed to use cases and liaisons
- Entities are returned in response DTOs and on most write operations

## Testing

- Entities themselves typically don't need separate tests
- Test entity creation and transformation in repository tests
