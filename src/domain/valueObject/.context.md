# Value Object Context

## Purpose

Value objects represent domain concepts with validation rules. They are immutable and guarantee type safety throughout the application.

## Critical Rules

- **Immutability**: Value objects cannot be modified after creation
- **Validation**: All constraints MUST be validated in the constructor (New\* function), mostly using regex and/or switch statements with exported prefixed variables (e.g., `ActivityRecordLevelDebug`, `ActivityRecordLevelInfo` etc)
- **Type safety**: Use specific types (e.g., `type IpAddress string`) instead of primitives
- **No business logic**: Value objects only validate and represent data; business operations go in use cases

## Constructor Pattern

All value objects MUST follow this pattern:

```go
type ValueObjectName string

func NewValueObjectName(value any) (vo ValueObjectName, err error) {
    // 1. Convert input to appropriate type using tkVoUtil
    stringValue, err := tkVoUtil.InterfaceToString(value)
    if err != nil {
        return vo, errors.New("ValueObjectNameMustBeString")
    }

    // 2. Validate business rules
    if stringValue == "" {
        return vo, errors.New("ValueObjectNameCannotBeEmpty")
    }

    // 3. Additional validations...

    // 4. Return validated value object
    return ValueObjectName(stringValue), nil
}
```

## Common Methods

- `String()`: Return string representation
- Conversion methods: Transform to related types (e.g., `ToCidrBlock()`) when appropriate

## Naming Conventions

- Constructor: `New{ValueObjectName}` (e.g., `NewIpAddress`)
- Error messages: `{ValueObjectName}{Constraint}` (e.g., "IpAddressCannotBeEmpty")
- Boolean methods: Use `Is`, `Has`, `Should` prefixes

## Testing Requirements

- MUST test all validation rules
- MUST test edge cases and boundary conditions
- MUST test error messages match expected format
- Use table-driven tests for multiple validation scenarios
