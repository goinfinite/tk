# Domain Layer Context

## Purpose

The domain layer is the core of the application, containing all business logic and rules. It is completely independent of infrastructure and presentation concerns.

## Architecture Principles

- **Zero external dependencies**: Domain layer must NOT depend on infra or presentation layers
- **Dependency inversion**: Repository interfaces are defined here; implementations are in infra layer
- **Pure business logic**: All domain rules and constraints are enforced here

## Layer Components

- **`entity/`**: Persistent business objects with identity (e.g., ActivityRecord)
- **`valueObject/`**: Immutable objects validated by business rules (e.g., IpAddress, Fqdn)
- **`dto/`**: Data transfer objects for communication between layers
- **`repository/`**: Interface definitions for data access (implemented in infra layer)
- **`useCase/`**: Business operations orchestrating entities and value objects

## Package Imports

- **Allowed**: Only other domain sub-packages
- **Forbidden**: `infra/*`, `presentation/*`, or any external libraries not essential for business logic

## Error Handling

- Return descriptive error strings (e.g., "IpAddressValueCannotBeEmpty")
- Common domain errors are defined in `src/domain/useCase/index.go`
- Never expose infrastructure details in error messages

## Testing

- All domain components MUST have comprehensive unit tests
- Tests should focus on business rule validation
- Use test cases (table-driven tests) wherever possible
