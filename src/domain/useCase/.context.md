# Use Case Context

## Purpose

Use cases orchestrate business operations by coordinating entities, value objects, and repository interactions. They encapsulate application-specific business rules, including setting default values when not provided or unable to be set by the caller.

## Critical Rules

- **Single responsibility**: Each use case does ONE thing
- **Repository injection**: Accept repository interfaces as parameters
- **DTO communication**: Accept and return DTOs
- **Business logic here**: Orchestration and business rules belong in use cases
- **No infrastructure**: Never depend on specific infrastructure implementations

## Use Case Pattern

```go
func ActionEntity(
    entityQueryRepo tkRepository.EntityQueryRepo,
    entityCmdRepo tkRepository.EntityCmdRepo,
    actionDto tkDto.ActionEntity,
) (responseDto tkDto.ActionEntityResponse, err error) {
    // 1. Validate business rules
    // 2. Coordinate with repositories
    // 3. Transform results
    // 4. Return DTO
    return responseDto, nil
}
```

## Special Cases

### Non-blocking Side Effects

Some use cases are fire-and-forget operations (e.g., activity logging):

```go
func CreateActivityRecord(
    activityRecordCmdRepo tkRepository.ActivityRecordCmdRepo,
    createDto tkDto.CreateActivityRecord,
) {
    err := activityRecordCmdRepo.Create(createDto)
    if err != nil {
        // Log but don't fail the caller's operation
        slog.Error("CreateActivityRecordInfraError", ...)
    }
}
```

## Common Use Case Types

- **Create**: Validate and persist new entities
- **Read**: Query and return entities with pagination
- **Update**: Modify existing entities
- **Delete**: Remove entities (validate constraints first)

## Error Handling

- Use descriptive error messages
- Common errors defined in `src/domain/useCase/index.go`
- Log infrastructure errors but return domain errors
- Use `slog.Error` for user-facing logs (no `slog.Any`)
- Use `slog.Debug` for development debugging (can use `slog.Any`)

## Naming Conventions

- Function: `{Action}{Entity}` (e.g., `CreateActivityRecord`, `ReadAccounts`)
- File: `{action}{Entity}.go` (e.g., `createActivityRecord.go`)

## Testing Requirements

When the use case is a simple pass-through to a repository, it does not need to be tested.

When the use case is a complex operation, it MUST be tested:

- MUST test business logic validation
- MUST test error conditions
- MUST test integration with repositories (using mocks or test repos)
- Use table-driven tests for multiple scenarios
