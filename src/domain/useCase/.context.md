Business operations that orchestrate entities and repositories. Each use case is a single function accepting repository interfaces and DTOs.

## Summary

- createActivityRecord.go — persists an activity record as a fire-and-forget side effect (errors logged, not returned)
- readActivityRecords.go — queries activity records via the query repo; defines default pagination and the `ErrActivityRecordNotFound` sentinel
- deleteActivityRecord.go — deletes an activity record via the cmd repo; wraps infra errors as domain errors
- honeypotSentinels.go — typed sentinel errors: `ErrNilHoneypotQueryRepo`, `ErrNilHoneypotCmdRepo`
- honeypotReadBanDecision.go — resolves ban tier from hit record with TTL check; accepts `HoneypotBanDuration` VO; returns sentinel error on nil repo
- honeypotCreateHit.go — increments hit count and probabilistically (~2%) enforces max entries; accepts `HoneypotMaxEntries` VO; no-op on nil repo
- honeypotReadStatsReport.go — delegates to query repo's ReadReport; accepts `HoneypotBanDuration` VO; returns empty report on nil repo
- honeypotRunMaintenance.go — accepts `RunHoneypotMaintenanceRequest` DTO; cleans expired entries, enforces max entries, produces stats report as activity record; skips stats when DB empty or repos nil; errors logged via slog.Error

## Guidance

- SHOULD accept repository interfaces as parameters for testability.
- SHOULD log infrastructure errors with `slog.Error` but return domain-level error messages.
- Use VOs for typed parameters (HoneypotBanDuration, HoneypotMaxEntries). Use DTOs for multi-param use cases.
