# DTO Context

## Purpose

DTOs are temporary structs used to transfer data between layers. They facilitate communication without coupling layers to specific implementations.

## Critical Rules

- **Temporary by design**: DTOs exist only during data transfer; they are not persisted
- **Use value objects**: All DTO fields should use domain value objects when possible
- **No business logic**: DTOs are pure data containers
- **Request/Response pattern**: Separate DTOs for requests and responses when needed

## Structure Guidelines

```go
type ActionName struct {
    RequiredField    tkValueObject.SomeType  `json:"requiredField"`
    OptionalField    *tkValueObject.SomeType `json:"optionalField"`
    PrimitiveField   any                     `json:"primitiveField"` // Only when VO doesn't exist
}
```

## Naming Conventions

- Request DTOs: `{Action}{Entity}` (e.g., `CreateActivityRecord`, `DeleteActivityRecord`)
- Response DTOs: `{Action}{Entity}Response` (e.g., `ReadActivityRecordsResponse`)
- Use descriptive action verbs: Create, Read, Update, Delete, etc.

## Field Ordering

1. Required fields first (non-pointer)
2. Optional fields second (pointer)
3. Within each group, order by importance then alphabetically

## JSON Tags

- All fields MUST have json tags for serialization
- Use camelCase for JSON field names
- Omitempty is generally NOT needed (clients should handle null)
- Internal fields can omit json tags

## Testing

- DTOs typically don't need separate tests
- Test usage of DTOs in use cases and liaisons instead
